✦ This is a sophisticated architectural question. You are touching on the divide
  between Application Logic (Python) and Database Logic (PostgreSQL).


  To a professional Database Engineer, moving queries into the database itself is
  called the Stored Procedure/Function approach. It is highly performant and secure
  because the API doesn't even need to know the table structure—it only knows how to
  "call" a specific operation.


  Here is how an expert would structure this using PostgreSQL Functions and
  Individual SQL Files while keeping your Python API clean.


  1. The Database Layer: db/functions/
  Instead of raw SQL in Python, we store the logic in the database. Create
  individual .sql files in your db/ folder. This allows you to version-control your
  queries.

  File: db/functions/update_scrub_stats.sql


   CREATE OR REPLACE FUNCTION update_scrub_stats(p_bytes_removed INTEGER)
   RETURNS VOID AS $$
   BEGIN
       UPDATE system_stats
       SET photos_cleaned = photos_cleaned + 1,
           bytes_removed = bytes_removed + p_bytes_removed,
           last_updated = CURRENT_TIMESTAMP
       WHERE id = (SELECT id FROM system_stats LIMIT 1);
   END;
   $$ LANGUAGE plpgsql;
  Note: You would add \i /path/to/functions/update_scrub_stats.sql to your init.sql
  so Docker loads it on startup.

  ---


  2. The API "Data Access" Layer: backend/api/db/
  Since you want the Python calls to live in the api folder but remain modular, we
  create a dedicated db/ subfolder inside api/. Each query gets its own Python file.

  File: backend/api/db/save_scrub.py


   from db_connection import get_db_connection

   def call_update_scrub_stats(bytes_removed: int):
       """
       Executes the stored procedure in Postgres.
       The Python code is now just a 'trigger'.
       """
       with get_db_connection() as conn:
           with conn.cursor() as cur:
               # We call the function by name, passing the argument
               cur.execute("SELECT update_scrub_stats(%s);", (bytes_removed,))
               conn.commit()

  File: backend/api/db/get_stats.py


   from db_connection import get_db_connection

   def call_get_system_stats():
       """
       Retrieves the data. If you had a 'get_stats' function in PG,
       you would call it here.
       """
       with get_db_connection() as conn:
           with conn.cursor() as cur:
               cur.execute("SELECT photos_cleaned, bytes_removed FROM system_stats
   LIMIT 1;")
               return cur.fetchone()

  4. Why this is the "Ultimate Pro" Move:


   1. Security: Your scrubber_app_user only needs EXECUTE permission on the
      functions. They don't even need UPDATE or SELECT permissions on the tables
      themselves. This prevents SQL injection and data leaks.
   2. Performance: PostgreSQL pre-compiles Stored Functions. They run faster than
      raw strings sent over the network.
   3. Maintainability: If you change your table schema (e.g., split system_stats
      into multiple tables), you only update the .sql file in the db/ folder. Your
      Python code remains exactly the same because it only cares about the function
      name update_scrub_stats.
   4. Clarity: In your file tree, you can see exactly what database operations exist
      just by looking at the filenames in backend/api/db/.
